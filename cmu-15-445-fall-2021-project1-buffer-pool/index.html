Summary

第1个项目是实现Buffer Pool Manager，BufferPoolManager 负责在磁盘和内存之间搬运物理页面的，如果页面被标记为脏，那么需要写回磁盘，保证数据不丢失，如果Buffer Pool 满了，那么需要从内存总换出页面到磁盘上，空出空闲位置给新的页面请求。实现在内存和磁盘之间搬运数据，对请求者透明。

Buffer Pool 项目的实现包括三个部分

  • Task 1: LRU Replacement Policy
  • Task 2: Buffer Pool Manager Instance
  • Task 3: Parallel Buffer Pool Manager

分别是实现LRU页面替换策略[1]，实现Buffer Pool Manager 的行为[2]（保证线程安全性，意味着Task2的实现必须加锁保护，否则通不过并发测试的），Task 3 主要是分配多个bufferpoolmanager 来减少锁竞争的[3]。

坑点

1、LRUReplacer::Unpin( )的时候是不需要将page调到LRU开头的，原地不动就好

2、BufferPoolManagerInstance::NewPgImp( ) newpage的时候，如果空闲页是从LRU返回的page，那么证明此page是被复用了，此时需要检查页面的is_dirty 标志位，如果为脏，需要写回磁盘，否则新页面一覆盖，这部分数据就丢失了。

3、BufferPoolManagerInstance::UnpinPgImp( ) 当所有线程都不 pin this Page时候, 就可以交给 LRU调度了，否则就内存泄露了，这个页没法再被复用

4、BufferPoolManagerInstance::UnpinPgImp( ) 的传入参数 is_dirty为true的时候再标记内存页，无脑标记是不可以的，可能覆盖掉原先页面的标记，导致数据丢失，因为你不标记，但是其他线程可能标记了，所有线程都能标记页面的

Task 1: LRU Replacement Policy

lru_replacer.h

LRU数据结构的组成就是 “list + hashmap” ，这两种数据结构的组合可以提供 O(1)时间复杂度的 add/remove/find 操作。C++里可以使用标准库的std::list 和 std::unordered_map来实现LRU。

LRU 算法可以参考小林coding的这篇文章——手撕 LRU 算法（更正版）

lru_replacer.cpp

接下来是lru_replacer的实现部分，

  • Victim——向LRU请求一个内存页，如果没有空闲页，则失败
  • Pin—表示将内存页pin在内存里，不可被LRU调度，所以将内存页踢出list_，这样就不会被LRU调度了
  • Unpin——表明可被 LRU调度, 加入 list_，(注意：如果内存页已经处于被LRU管理中，那么不需要将此页掉到list_开头，此处的行为常说的LRU算法不太一样)
  • Size——返回LRU管理的内存页的数目
